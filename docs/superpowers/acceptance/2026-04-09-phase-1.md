## Why 4a / 4b are deferred, not failed

The Phase 1 design said "anonymous requests for now." That assumption holds for read endpoints (`GET /api/sales`) but fundamentally breaks for write endpoints (`POST /api/sales`), which yardsailing protects with `Depends(get_current_user)` plus a payment-token requirement after the first sale.

Anonymous create-yard-sale is **architecturally impossible** without one of:
- Modifying yardsailing to accept JAIN-issued credentials (rejected — design said yardsailing stays unchanged)
- Adding real auth to JAIN (out of Phase 1 scope by the design)

Rather than skip the criterion or fake it, Phase 1 ships with **Layer 1 auth-aware Jain**: per-plugin auth state flows through the chat context, and the `create-sale` skill is structured so the very first thing Jain does is check `[auth state: yardsailing=...]`. If not authenticated, Jain politely tells the user to log in via Settings instead of attempting the broken tool call.

The end-user experience is therefore **graceful and correct** — Jain knows its limits, communicates them clearly, and never produces opaque API errors. The "real" create flow ships in Phase 2 along with OAuth.
