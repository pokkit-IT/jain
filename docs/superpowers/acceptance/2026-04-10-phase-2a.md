## Deviations from the plan

During acceptance, three issues surfaced that the plan hadn't anticipated. Each was fixed during the acceptance run and committed as `aad237b fix(mobile): make google sign-in actually work end-to-end`:

1. **`useProxy: true` is dead** — Expo's auth proxy service at `auth.expo.io` was shut down in February 2024, but the plan assumed it still worked. The `useProxy: true` flag is now a no-op and `makeRedirectUri({ useProxy: true })` returns a useless custom scheme. Fix: drop `useProxy` entirely; use platform-specific client IDs (`iosClientId`, `webClientId`, etc.) and let `expo-auth-session` generate the right redirect per platform.

2. **`response_type` defaulted to `token`, not `id_token`** — the Google provider defaults to returning a Google access token (for calling Google APIs), not an OIDC ID token (a JWT signed by Google for identity). JAIN's backend needs the ID token. Fix: explicitly set `responseType: ResponseType.IdToken`.

3. **`expo-secure-store` is native-only** — throws on web. Fix: `tokenStorage.ts` now detects `Platform.OS === 'web'` and falls back to `window.localStorage`. Less secure than iOS Keychain but acceptable for dev testing.

The plan's Task 12 (`googleAuth.ts` implementation) was wrong in subtle ways because it was written against older Expo AuthSession documentation. A future plan touching OAuth should validate the Expo + Google pairing against the current SDK's actual behavior before committing code, not against the historical docs.

## Known limitation (deferred to Phase 2A.1)

**Google Sign-In does not work in Expo Go on iOS** because:

1. Expo Go's native binary has a fixed `Info.plist` — you cannot register custom URL schemes (like the reversed client ID `com.googleusercontent.apps.<reversed>`) required for native iOS OAuth callbacks
2. The Expo auth proxy (which used to paper over this) is gone
3. The only workable path on iOS is a **development build** via EAS Build with a custom bundle identifier, which requires either a Mac for local iOS Simulator builds or Apple Developer Program ($99/yr) for TestFlight distribution

**Phase 2A.1 should address this** by either:
- Setting up EAS Build with a real bundle identifier (`com.jshelly.jain` or similar)
- Creating a Phase 2A.1-specific iOS OAuth client in Google Cloud Console with that bundle ID
- Distributing via TestFlight or local device install
- Replacing Expo Go with the development build as the default iOS test surface

Alternatively, Phase 2A.1 could be skipped entirely and iPhone Google Sign-In deferred until Phase 2C (Apple Sign In), since iOS users can use Apple Sign In as their primary login path.
